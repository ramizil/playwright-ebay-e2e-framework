eBay E2E
Automation Framework

A production-grade end-to-end test framework that automates eBay's shopping workflow — from search and filtering to product selection and cart validation. Built with Python 3.12, Playwright, and pytest, this framework showcases how to architect a maintainable, scalable, and resilient test suite that teams can confidently run in CI/CD pipelines across multiple browsers.

The project implements a five-layer architecture — Tests, Pages, Core Engine, Utilities, and Configuration — with clean separation of concerns so that each layer can evolve independently. Smart locators with automatic fallback strategies ensure tests survive UI changes, while exponential-backoff retry logic handles the flakiness inherent in real-world web applications. Test data lives in external JSON files, so new scenarios can be added without touching a single line of test code.

02 — Smart Locators

Resilient Element Location

Every UI element is defined with at least 2 locator strategies. When the primary fails (DOM change, A/B test, layout shift), the framework automatically falls back — keeping tests green without manual fixes.

# Every element gets ≥ 2 strategies SEARCH_INPUT = SmartLocator( name="search_input", strategies=[ LocatorStrategy("css", "input.gh-tb[type='text']", "by class"), LocatorStrategy("xpath", "//input[@name='_nkw']", "by name attr"), ], )
# BasePage.find_element() — the fallback engine def find_element(self, smart_locator, timeout=None): for idx, strategy in enumerate(smart_locator.strategies): try: locator = self._create_playwright_locator(strategy) locator.wait_for(state="visible", timeout=timeout) self.logger.info("✓ Strategy #%d succeeded", idx+1) return locator except TimeoutError: self.logger.warning("✗ Strategy #%d failed", idx+1) self.take_screenshot(f"FAIL_{smart_locator.name}") raise SmartLocatorError("All strategies exhausted")

Recommended Priority Order

Strategies should be ordered by resilience to change — user-facing attributes survive refactors far better than DOM structure selectors. Playwright's own docs advocate this same hierarchy: prefer locators that mirror how real users perceive the page.

PriorityMethodPlaywright APIWhy This RankExample
1st test_id page.get_by_test_id() Explicitly added for testing — immune to styling, layout, and copy changes ux-call-to-action
2nd role page.get_by_role() Reflects accessibility semantics — survives CSS refactors and DOM restructuring button, name=Search
3rd placeholder page.get_by_placeholder() Tied to UX copy for inputs — stable unless placeholder text is rewritten Search for anything
4th label page.get_by_label() Bound to form labels — stable and accessibility-friendly Email address
5th text page.get_by_text() User-visible content — can break with i18n or marketing copy changes Add to cart
6th css page.locator(selector) Depends on class names and IDs — can break with CSS-in-JS or framework updates input#gh-btn
7th xpath page.locator("xpath=…") Most brittle — tightly coupled to DOM tree structure and element hierarchy //input[@name='_nkw']
Most resilient — use as primary
Good stability — solid secondary choice
Context-dependent — useful as fallback
Last resort — most fragile

In practice: eBay doesn't expose data-testid attributes on most elements, so this framework uses CSS or role as the primary strategy and XPath as the fallback — the best available options when you don't control the application under test.

03 — Resilience

Retry with Exponential Backoff

The @with_retry decorator wraps any function with configurable retry logic. Handles transient failures from network hiccups, slow DOM renders, and overlay animations.

@with_retry(max_attempts=3, backoff_factor=1.0, jitter=True) def _click_add_to_cart(self): element = self.find_element(self.ADD_TO_CART_BUTTON) element.click()

Backoff Formula

wait = backoff_factor × 2^(attempt-1) ± 25% jitter

AttemptWaitCumulativeAction
10s0sFirst try — immediate
2~1s~1sRetry after short pause
3~2s~3sFinal attempt + screenshot on fail
🔄

Decorator Form

@with_retry() wraps methods permanently. Used in BasePage for click, fill, type actions.

🎯

Inline Form

retry_call(func, ...) for one-off retries without decorating the function.

07 — Parallel Execution

Speed at Scale

Two strategies for parallel execution — choose based on your environment.

⚡

pytest-xdist

Multi-process parallelism within a single machine. Each worker gets an isolated BrowserContext — no shared cookies or state.

$ pytest -n auto --dist=loadfile
🐳

Docker Compose

Separate containers per browser. Chromium, Firefox, and Edge run simultaneously with their own Allure output.

$ docker-compose up --build

Multi-Browser & Channel Support

Browser profiles map friendly names to Playwright engines and optional channels. Channels launch branded builds (Chrome, Edge) instead of the bundled open-source Chromium — giving you real-world coverage across browser vendors and version tracks.

Session Isolation

Each test runs in a completely isolated browser session — no driver sharing between parallel tests. This is enforced by Playwright's context model.

BoundaryImplementationScope

Separate Reports per Run

Every test invocation generates a unique run ID (timestamp-based, or set via EBAY_RUN_ID). Allure results are written into a timestamped sub-folder so previous results are never overwritten.
