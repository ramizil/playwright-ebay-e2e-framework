eBay E2E Framework Runbook

A plain-English guide to every folder, class, and function in this project — and how they all fit together.

🎯
What does this project do?

It automatically opens a web browser, goes to eBay.com, searches for products, filters by price, adds items to the shopping cart, and then checks that the cart total is within budget. It does all this without any human clicking — the code drives the browser.

Think of it as a robot shopper that tests whether eBay's website is working correctly.

🧰

📁 core/ — The Engine

This is the heart of the framework. It provides the building blocks that every page object uses.

smart_locator.py — Multi-Strategy Element Finder

Core idea: Instead of relying on one CSS selector that might break when eBay redesigns their site, we define multiple ways to find each element. If the first way fails, we try the second, then the third, and so on. Strategy #1 is chosen based on the element's identifier type (stable ID, auto-generated ID, or none), and Strategy #2 is always a relative XPath.

class LocatorStrategy (frozen dataclass)

A single way to find an element. Has three fields:

FieldDescriptionExample
methodThe type of locator"css", "xpath", "role", "text", "test_id", "placeholder", "label"
valueThe selector expression"input#gh-ac", "//button[@id='search']"
descriptionHuman-readable label for logs"search input by ID"
class SmartLocator (dataclass)

A group of strategies for one UI element. Requires at least 2 strategies (so there's always a fallback).

# Tier 1 — Stable ID first, relative XPath fallback
SEARCH_INPUT = SmartLocator(
    name="search_input",
    strategies=[
        LocatorStrategy("css", "#gh-ac", "by stable ID"),
        LocatorStrategy("xpath", "//input[@name='_nkw' and @type='text']", "by XPath attr"),
    ],
)

Properties:

  • .primary → first strategy (the one we expect to work)
  • .fallbacks → all strategies except the first

Tiered Strategy — Choosing What Goes in Strategy #1

Element Has…Strategy #1 (Primary)Strategy #2 (Fallback)Examples
Stable ID
Human-authored, long-lived		 css by #id Relative XPath by attribute #gh-ac, #gh-btn, #atcBtn_btn_1, #qtyTextBox
Auto-generated ID
Dynamic, changes across builds		 css by aria-label, class, or data-testid Relative XPath by attribute input[aria-label*='Minimum Value'], [data-test-id='SUBTOTAL']
No ID at all
Only class names or text		 role or css by attribute Relative XPath role: button, name=Submit, a.pagination__next

Key rule: Never use an auto-generated ID (like s0-28-9-0-1[3]-x-btn) as Strategy #1 — it will break on the next deploy. Use aria-label or data-testid instead.

retry_handler.py — Never Give Up (Easily)

Some actions fail intermittently (network hiccup, slow page load). The retry handler automatically re-attempts with increasing delays between tries.

func @with_retry — Decorator 
@with_retry(max_attempts=3, backoff_factor=1.0, jitter=True)
def _click_add_to_cart(self):
    element = self.find_element(self.ADD_TO_CART_BUTTON)
    element.click()

If _click_add_to_cart throws an exception:

  1. Wait ~1 second (with random jitter)
  2. Try again
  3. If fails, wait ~2 seconds, try again
  4. If still fails after 3 attempts, give up and raise the error
ParameterDefaultMeaning
max_attempts3Total tries before giving up
backoff_factor1.0Base delay multiplier (seconds)
jitterTrueAdd randomness so retries don't all hit at once
func retry_call() — Standalone retry

Same as the decorator but for calling any function on the fly: retry_call(some_func, arg1, arg2, max_attempts=5)

logger_config.py — Structured Logging

Sets up Python's logging to write to both the console and a rotating file (logs/framework.log). Every page object, fixture, and utility gets its own named logger.

func setup_logging(level="INFO")

Called once at startup. Configures the root logger with a console handler (coloured) and a file handler (rotates at 5 MB, keeps 3 backups).

func get_logger(name) → Logger

Returns a named logger. Usage: logger = get_logger(__name__). The name appears in log output so you know which file produced each line.

Clickable links in PyCharm: Log lines include source references in Python traceback format (File "path", line N), so PyCharm automatically makes them clickable — jump straight to the code that produced each log entry.

📁 pages/ — Page Object Model

Each page on eBay gets its own Python class. The class knows where every button and input is (via SmartLocators) and how to interact with them (via methods).

Page Object Model (POM) is a design pattern: test code never touches CSS selectors or Playwright directly. Instead it calls methods like home.search("laptop"). If eBay changes their HTML, you only fix the locators in one place.

This package sits between the page objects and the test scenarios. It contains the four central business functions — each orchestrating one or more page objects into a complete user flow.

📁 tests/ — Test Files & Fixtures

conftest.py — Browser Fixtures (The Plumbing)

This file sets up and tears down the browser for each test. pytest automatically discovers fixtures defined here.

What's a fixture? A function that provides something a test needs (a browser, a page, settings). pytest injects it automatically when a test declares it as a parameter.

Session-scoped fixtures

FixtureScopeWhat it provides
test_settingssessionThe Settings object (loaded once, shared across all tests)
playwright_instancesessionStarts Playwright, yields it, then stops it when all tests are done
browser_profilesessionResolves EBAY_BROWSER env var to a BrowserProfile
browsersessionLaunches the actual browser (Chrome/Firefox/Edge). Kills lingering background processes for Edge/Chrome before launch. One browser for the whole session.

Function-scoped fixtures (single-function tests)

FixtureScopeWhat it provides
contextfunctionNew browser context per test (isolated cookies, storage). Enables tracing if configured.
pagefunctionNew page (tab) per test. This is what page objects use to interact with the site.

Class-scoped fixtures (multi-step flow classes)

FixtureScopeWhat it provides
class_contextclassShared browser context for all step methods in a test class. Cart, session, and cookies persist across steps — like TestNG's shared instance.
class_pageclassShared page (tab) for all step methods. This is what the TestNG-style test classes use.
scenario_dataclassReceives the parametrised scenario dict when using @pytest.mark.parametrize("scenario_data", ..., indirect=True) on a test class.

Why two scopes? Simple, single-function tests get a fresh page per test (full isolation). Multi-step class-based flows share a class_page across all step methods (so browser state — like cart contents — persists between steps).

hook pytest_runtest_setup — Step dependency skip (dependsOnMethods)

When a step method inside a test class fails, all subsequent step methods in the same class are automatically skipped — equivalent to TestNG's dependsOnMethods. Tracked per parametrised variant so failures in [headphones_under_50] don't affect [phone_case_under_20].

hook auto_screenshot_on_failure (autouse)

Runs after every test step. If the step failed, it captures a screenshot and saves the Playwright trace file. Works for both function-scoped and class-scoped tests.

test_shopping_flow.py — TestNG-Style Test Class

Inspired by Java/TestNG: Each test flow is a class where every step is a separate, numbered method — like @Test(priority=0, description="..."). If a step fails, all dependent steps are automatically skipped (equivalent to dependsOnMethods). Shared state flows through a FlowState dataclass — similar to Java instance fields like this.urls.

@pytest.mark.parametrize("scenario_data", _scenarios, indirect=True)
class TestEbayShoppingFlow:
    """
    Description:
        Automated shopping flow: search, collect, add to cart, validate.

    Steps to reproduce (manual):
        1. Open eBay and sign in
        2. Search for products by keyword
        3. Filter by "Buy It Now" and price range
        4. Add qualifying items to cart
        5. Verify cart subtotal ≤ budget

    Expected:
        Cart total does not exceed the calculated budget.
    """

    # ── @BeforeClass ──
    @pytest.fixture(autouse=True, scope="class")
    def setup_flow(self, request, class_page, test_settings, scenario_data):
        request.cls.page = class_page
        request.cls.settings = test_settings
        request.cls.scenario = scenario_data
        request.cls.state = FlowState()

    # ── @Test(priority=0, description="Sign in to eBay") ──
    def test_step_00_login(self): ...

    # ── @Test(priority=1, dependsOn="login") ──
    def test_step_01_search_and_collect(self): ...

    # ── @Test(priority=2, dependsOn="search") ──
    def test_step_02_add_to_cart(self): ...

    # ── @Test(priority=3, dependsOn="add_to_cart") ──
    def test_step_03_validate_cart(self): ...

Each step calls business_steps functions which orchestrate the page objects:

How-To: Add a New Page Object

  1. Create pages/my_page.py
  2. Import BasePage, SmartLocator, LocatorStrategy
  3. Define SmartLocators using the tiered strategy: stable ID → CSS #id; auto ID → CSS by aria-label/data-testid; no ID → role/class. Always add a relative XPath fallback.
  4. Write methods that call self.click(), self.fill(), self.get_text(), etc.
  5. Export from pages/__init__.py
from core.base_page import BasePage
from core.smart_locator import LocatorStrategy, SmartLocator

class MyPage(BasePage):
    # Tier 1: Stable ID → CSS #id + relative XPath fallback
    MY_BUTTON = SmartLocator(
        name="my_button",
        strategies=[
            LocatorStrategy("css", "#my-btn", "button by stable ID"),
            LocatorStrategy("xpath", "//button[@aria-label='My Button']", "button by XPath attr"),
        ],
    )

    # Tier 3: No ID → role-based + relative XPath fallback
    SUBMIT_BTN = SmartLocator(
        name="submit_button",
        strategies=[
            LocatorStrategy("role", "button, name=Submit", "submit by role"),
            LocatorStrategy("xpath", "//button[@type='submit']", "submit by XPath attr"),
        ],
    )

    def click_my_button(self):
        self.click(self.MY_BUTTON)

How It All Fits Together — The Full Picture

Startup Sequence (TestNG-style class flow)

1. You run: pytest tests/ -v
2. pytest discovers conftest.py (root) → calls pytest_configure()
   → setup_logging()           # configure console + file logging
   → load_settings()           # read config.yaml + env overrides
   → write_allure_environment()  # create environment.properties
   → init flow-tracking state on config:
     _failed_class_nodes = set()
     _flow_statuses = {}
     _flow_errors = {}

3. pytest discovers tests/conftest.py → registers fixtures
4. pytest discovers test classes → parametrizes from JSON

5. For each TestEbayShoppingFlow[headphones_under_50]:
   a. Class-scoped fixture chain fires (≈ @BeforeClass):
      test_settings → playwright_instance → browser_profile
      → browser (launches Chrome/Firefox/Edge)
      → class_context (shared cookies/state for ALL steps)
      → class_page (shared browser tab)
      → setup_flow (assigns self.page, self.settings, self.state)

   b. Step methods run in order:

      test_step_00_login(self)         ← @Test(priority=0)
        → login(self.page)

      test_step_01_search_and_collect  ← @Test(priority=1, dependsOn="login")
        → search_items_by_name_under_price()
           → HomePage(page).open() → search()
           → SearchResultsPage(page).filter_buy_it_now()
           → .apply_price_filter() → .collect_items_under_price()
        → self.state.urls = [collected URLs]

      test_step_02_add_to_cart         ← @Test(priority=2, dependsOn="search")
        → add_items_to_cart(page, self.state.urls)
           → for each URL: ProductPage → select_random_variants()
             → _click_add_to_cart() (@with_retry) → screenshot
        → self.state.added_count = N

      test_step_03_validate_cart       ← @Test(priority=3, dependsOn="add_to_cart")
        → assert_cart_total_not_exceeds(page, budget, added_count)
           → CartPage(page).open_cart() → assert total ≤ budget

      If ANY step fails:
        → auto_screenshot_on_failure captures evidence
        → parent_id added to _failed_class_nodes
        → ALL subsequent steps are SKIPPED (dependsOnMethods)

   c. Teardown (≈ @AfterClass):
      → setup_flow yield: collector.finish_scenario(status, error)
      → class_context closes (tracing saved)
      → HTML report generated for this scenario

6. Next scenario (phone_case_under_20) starts fresh
7. After all scenarios: browser closes, Playwright stops
8. Run summary HTML report generated
